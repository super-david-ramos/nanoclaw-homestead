## DEC-skill-resolution — user > role > shared skill resolution as a container-skill convention

- **source:** `docs/plan/decisions/01-skill-resolution.md`
- **status:** LOCKED (Accepted; shipped 2026-04-25)
- **scope:** container skills, skill resolution, role-resolver SKILL.md, agent group skills layout, session-manager sender/role projection, `inbound.db` message envelope
- **decision:**
  - Implement homestead-ts user > role > shared full-shadowing skill resolution on nanoclaw v2 via a single container convention skill (`container/skills/role-resolver/SKILL.md`) plus a three-directory layout per agent group: `groups/<group>/skills/{users/<user_id>,roles/<role>,shared}/<skill-name>/SKILL.md`.
  - First-match priority `users/<id>` → `roles/<role>` → `shared`. Activate exactly one matching skill.
  - All three trees mounted into the session container alongside standard nanoclaw container skills.
  - Sender + role visibility for the agent must surface in the session system prompt or message envelope.
- **consequences:**
  - Empty per-group `groups/<group>/skills/{users,roles,shared}/` scaffolding gets created during `/setup` for each agent group that wants the shadowing behavior.
  - Convention-only is sufficient for v0 (≤ 4 users, ~20 skills). Host-side pre-filtering (forked session-manager) is the next move only if convention drifts.
- **open detail:** Project sender→role into `inbound.db` at session wake (preferred; small touch to `src/session-manager.ts`) vs. stuff sender + role into `messages_in.content` or a `system`-kind pre-message (smaller change, uglier). Tracked in `06-open-questions.md` #1.

## DEC-voice — voice (Whisper/Kokoro) wraps every text channel

- **source:** `docs/plan/decisions/03-voice.md`
- **status:** LOCKED (Accepted; resolved 2026-05-01 with detailed execution model)
- **scope:** voice I/O, STT (Whisper), TTS (Kokoro), channel adapters, Telegram voice messages, delivery layer, `users.prefers_voice_replies`, `skill/voice-transcription`, `skill/local-whisper`, `skill/voice-in`, `skill/voice-out`, host process execution model, `data/models/ggml-base.bin`
- **decision:**
  - Voice on every channel via channel-I/O wrapping (STT inbound, TTS outbound). Not a separate channel branch.
  - **Inbound STT:** host-side, in-process, shell-out per call. Host calls `whisper-cli` (Homebrew `whisper-cpp`, Metal-accelerated bottle on Apple Silicon) via `execFile`. Audio normalized to 16 kHz mono WAV by `ffmpeg` first. Hook lives in each channel adapter, between platform fetch and `inbound.db` write. Transcript attached as a structured field on the message; original audio path preserved.
  - **Outbound TTS:** host-side; same shape. Trigger via delivery-layer check on `users.prefers_voice_replies`.
  - **Containers see no voice infra** — only the resulting text + an attachment reference.
  - **Default model:** `ggml-base.bin` (147 MB) under `data/models/`. Switchable via env.
  - **Per-user voice preference:** new column `users.prefers_voice_replies INTEGER NOT NULL DEFAULT 0`. Migration is a single `ALTER TABLE`. Read site: `delivery.ts` immediately before adapter `send`.
- **rationale (rejected alternatives):**
  - Per-container Whisper: rejected — 200–300 MB resident competes with agent runtime memory on 8 GB Mac Mini M2; bloats every image.
  - Long-lived host service: rejected — one-shot subprocess, daemon = `execFile` with extra ceremony.
  - Markdown-based voice preference: rejected — delivery layer needs programmatic, microsecond decision; LLM-interpreting markdown is too slow + non-deterministic.
- **resource expectation:** ~1 s wall per 30 s of audio on M1+ with `base`. Serializes if multiple voice notes arrive simultaneously (acceptable at homestead scale).
- **upstream origin:** Borrows brew formula choice + `whisper-cli` + `ffmpeg` invocation pattern from upstream's v1 `skill/voice-transcription` / `skill/local-whisper` (on `qwibitai/nanoclaw-whatsapp` fork repo, not directly portable to v2). Built fresh against v2's adapter + DB pipeline.

## DEC-no-pool-no-max — no pool-aware substitution, no Claude Code CLI / Max OAuth

- **source:** `docs/plan/decisions/04-no-pool-no-max.md`
- **status:** LOCKED (treated as Accepted per fork CLAUDE.md — decisions/ directory is load-bearing; no explicit `Status:` frontmatter)
- **scope:** model-routing strategy, Claude Agent SDK runtime, OneCLI Agent Vault credential injection, provider abstraction (`src/providers/`), OpenCode / Ollama / OpenRouter via skills, Max-subscription billing path, pool-snapshot ingestion, substitution router
- **decision:**
  - v0 uses nanoclaw's default path: Claude Agent SDK in-container + OneCLI Agent Vault for credential injection (`CLAUDE_CODE_OAUTH_TOKEN` from `claude setup-token`, or `ANTHROPIC_API_KEY`).
  - Provider abstraction lives in nanoclaw's `src/providers/`, not a homestead-side adapter. OpenCode / Ollama / OpenRouter are installed via existing skills (`/add-opencode`, `/add-ollama-provider`).
- **what we will NOT build:**
  - Pool-snapshot ingestion layer (Anthropic dashboard scrape, OCR fallback).
  - Substitution router (Opus → Sonnet → Haiku at thresholds).
  - Model-routing config DSL.
  - Pluggable agent-runtime adapter swapping Claude Code CLI in for Max billing.
- **revisit trigger:** model-routing cost becoming a real bottleneck (daily spend > $X sustained, or Max sub becomes worth optimizing for). Write a fresh decision doc on revisit; don't append to this one.

## DEC-self-improving — self-improving skills via propose-and-confirm (informed by Hermes)

- **source:** `docs/plan/decisions/05-self-improving.md`
- **status:** LOCKED (Accepted; shipped 2026-04-26)
- **scope:** self-improving skills, `auto-skill-save` container skill, Hermes substrate evaluation, approval gate, role-based skill resolution, Phase 0 defaults
- **decision:**
  - Stay on nanoclaw; port the *capability* (post-complex-turn skill proposal), not the Hermes substrate.
  - Built `container/skills/auto-skill-save/SKILL.md` with the following defaults (opposite to Hermes):
    - **Eligibility check first.** ≥ 5 tool calls is necessary but not sufficient — must also be a named, generalizable workflow with no volatile-state dependency.
    - **Approval before write.** `ask_user_question` blocks the turn; user picks Save / Edit first / Different scope / Skip. Never write a SKILL.md silently.
    - **Three-tier writes.** Default `shared/`. Bumps to role or user only on explicit choice. Plays clean with role-resolver convention (DEC-skill-resolution).
    - **Hard skip for child/guest roles.** If `sender_role` ∈ `{child, guest}`, skill bails before eligibility check.
    - **Off by default in Phase 0.** Container skill on disk and discoverable; eligibility check tightens to "explicit user request only" until a few real reusable workflows exist.
- **anti-patterns rejected (Hermes defaults):** no approval gate, single shared `~/.hermes/skills/`, per-turn latency cost from hardcoded post-task prompt.
- **reintroduction trigger for looser auto-trigger:** 3+ workflows the household has actually re-invoked manually that *would* have been better as skills.

## DEC-obsidian-markdown — Obsidian-compatible markdown for the memory surface

- **source:** `docs/plan/decisions/06-obsidian-markdown.md`
- **status:** Accepted (unlocked — no explicit `Status:` field; matches ADR pattern in form and content)
- **scope:** `groups/<folder>/`, memory surface, Obsidian vault, PARA folders (projects, areas, resources, archive, conversations), GFM markdown, wikilinks, `global/CLAUDE.md`
- **decision:**
  - Make nanoclaw's per-group markdown folders cleanly Obsidian-compatible. Group folders ARE the vault — no sync layer.
  - **PARA-ish layout per agent group** under `groups/<group-folder>/`:
    - `CLAUDE.md` (entry point — nanoclaw existing convention)
    - `projects/` — active multi-step efforts
    - `areas/` — ongoing responsibilities
    - `resources/` — reference notes
    - `archive/` — past projects, completed reminders
    - `conversations/` — searchable transcripts (existing nanoclaw convention)
  - **Plain GFM only in stored notes.** Slack mrkdwn / Telegram-flavored markdown applies only to outbound messages on those platforms; stored markdown stays GFM for Obsidian.
  - **No vault-sync layer.** Point Obsidian directly at `~/Code/nanoclaw-homestead/groups/family/` (or per household).
  - **`global/CLAUDE.md` is shared facts**, written only by the main agent group. Treat like an Obsidian "kitchen sink" note for cross-group facts.
  - **Wikilinks `[[...]]` are fine** — Obsidian-native, ignored by stock markdown renderers.
- **what this decision does NOT include:** vault-sync daemon, separate "vault" filesystem outside `groups/`, custom markdown extension only readable inside the agent.
